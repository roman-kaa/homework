# Домашнее задание к занятию "`Резервное копирование`" - `Калинковский Роман`



Резервное копирование — процесс создания копии данных для их последующего восстановления в случае сбоя оборудования, 
ошибок пользователей, программных ошибок или иных аварийных ситуаций.

Основные виды резервного копирования:

Полное (Full Backup) — копируются все данные.
Инкрементное (Incremental Backup) — сохраняются только изменения с момента последнего резервного копирования.
Дифференциальное (Differential Backup) — сохраняются изменения с момента последнего полного бэкапа.
Снимки файловой системы (Snapshots) — моментальные снимки состояния данных.
PITR (Point-In-Time Recovery) — восстановление на конкретный момент времени.
Репликация (Replication) — поддержание актуальной копии БД на другом сервере.


### Задание 1 Резервное копирование

Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования.

Необходимо описать, какие варианты резервного копирования подходят в случаях:

1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.

1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных.

`Приведите ответ в свободной форме........`

### 1.1. 
Необходимо восстанавливать данные в полном объёме за предыдущий день
Для данной задачи наиболее подходит:
Полное ежедневное резервное копирование
Схема рабоы:

Каждый день ночью выполняется полный бэкап базы данных.
При аварии данные восстанавливаются из последней резервной копии.

```Text
Время	  Действие
------|--------------------------------
01:00	  Создание полного бэкапа
15:00	  Сбой БД
15:10	  Восстановление из бэкапа 01:00
```
Преимущества:

Простота реализации.
Простое восстановление.
Высокая надёжность.

Недостатки:

Большой объём хранимых данных.
Длительное создание резервной копии.

Для крупных БД часто используют комбинацию:

еженедельный Full Backup;
ежедневный Incremental Backup.

Тогда восстановление происходит из полного бэкапа плюс всех накопленных инкрементов.


### 1.2. 
Необходимо восстанавливать данные за час до предполагаемой поломки

В данном случае обычного полного резервного копирования недостаточно.

Подходящее решение:

Point-In-Time Recovery (PITR)

PITR позволяет восстановить базу данных на конкретный момент времени.

Принцип работы:

Выполняется полный бэкап.
Параллельно сохраняются журналы транзакций:
PostgreSQL — WAL (Write-Ahead Log);
MySQL — Binary Log.
При аварии:
разворачивается последний полный бэкап;
затем проигрываются журналы транзакций до нужного времени.

Пример:

```Text
Время	  Событие
------|--------------------------
00:00	  Полный бэкап
13:00	  Ошибочное удаление данных
14:00	  Обнаружение проблемы
```
Восстановление выполняется на время:

12:59:59

В результате потеря данных будет минимальной. WAL-архивирование в PostgreSQL позволяет восстанавливать состояние БД практически на любую секунду между резервной копией и моментом аварии.

Преимущества:

Минимальная потеря данных.
Возможность откатиться до любого момента времени.

Недостатки:

Более сложная настройка.
Необходимость хранения журналов транзакций.


### 1.3.
Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных?

Да, такой сценарий широко используется в высоконагруженных системах.
Репликация и Failover


```Text
Схема:

Primary (Master)
       |
       |
       V
Replica (Standby)
```

Основной сервер постоянно передаёт изменения на реплику.

Если основной сервер выходит из строя:

```Text
Master X (отказ)

      ↓

Replica → становится новым Master
```

Такой процесс называется:

Failover (автоматическое переключение);
High Availability (HA).

Используемые технологии:

Для PostgreSQL:

Streaming Replication;
Patroni;
Pgpool-II;
repmgr.

Для MySQL:

Replication;
MySQL InnoDB Cluster;
Group Replication;
MHA.

Преимущества:

Минимальный простой.
Быстрое восстановление сервиса.
Пользователи могут даже не заметить отказа.

Недостатки:

Более сложная инфраструктура.
Дополнительные серверы.

Важно понимать:

### Репликация не заменяет резервное копирование. Если пользователь удалит данные на основном сервере, это удаление будет передано и на реплику. Поэому репликация и резервное копирование должны использоваься совместно!!!

### Задание 2 PostgreSQL

`Приведите ответ в свободной форме........`

2.1. Пример команды резервирования данных

Утилита PostgreSQL:

```
pg_dump
```
Пример создания резервной копии:

pg_dump -Fc mydb > db.dump

где:

-Fc — создание резервной копии в специальном архивном формате;
mydb — имя базы данных;
db.dump — файл резервной копии.
Пример восстановления базы

Утилита:

pg_restore

Создание новой БД:

createdb newdb

Восстановление:

pg_restore -d newdb db.dump

или восстановление поверх существующей БД:

pg_restore -d postgres --clean --create db.dump

Параметры:

--clean — удалить существующие объекты;
--create — создать базу заново перед восстановлением.
2.1.* Возможно ли автоматизировать этот процесс?

Да.

На практике резервное копирование почти всегда автоматизируется.

В Linux через cron

Пример задания:

0 2 * * * pg_dump -Fc mydb > /backup/mydb_$(date +\%F).dump

Что происходит:

каждый день в 02:00;
создаётся новый файл резервной копии.
Bash-скрипт

Пример:

#!/bin/bash

DATE=$(date +%F)

pg_dump -Fc mydb > /backup/mydb_$DATE.dump

find /backup -type f -mtime +30 -delete

Скрипт:

создаёт бэкап;
удаляет копии старше 30 дней.
Специализированные решения

Для PostgreSQL часто используют:

pgBackRest
Barman
WAL-G

Они обеспечивают:

автоматические бэкапы;
хранение WAL-журналов;
PITR;
резервирование в облако;
контроль целостности копий.
